# 知识回写规则

## 触发时机

最终一致性检查全部 PASS 后，**必须执行**知识回写，不可跳过。

## 回写原则

**只记录可复用经验，不记录项目特有数据**（零件号、项目人名、具体日期等不写入知识库）。

## 步骤 1：OEM 知识回写

检查 `knowledge/oem/{oem名}.md` 是否存在：
- **不存在** → 读取 `knowledge/oem/_oem-template.md`，按模板结构创建新文件
- **已存在** → 读取现有文件，合并新发现（不覆盖已有内容，只补充）

必须回写的内容（从本次项目中提取可复用经验）：

| 章节 | 回写内容 | 数据来源 |
|------|---------|---------|
| 文档体系 | 文档编号前缀含义、文档层级结构、覆盖规则 | 文档解析阶段文件分类 |
| 术语惯例 | OEM 特有术语 ↔ 通用用语对照 | 全程积累 |
| 交付物命名惯例 | TDR 中的交付物编号与名称映射 | 文档解析阶段交付物清单 |
| 客户模板藏匿位置 | 哪些嵌入文件是模板、在哪层发现的 | 递归提取 + 模板检测 |
| 嵌入文件结构树 | 完整的嵌入文件层级树 | 递归提取 |
| 处理注意事项 | 本次处理中踩过的坑、区域差异、特殊要求 | 全程积累 |

## 步骤 2：零部件知识回写

检查 `knowledge/component/{零部件类型}.md` 是否存在：
- **不存在** → 读取 `knowledge/component/_component-template.md`，按模板结构创建新文件
- **已存在** → 读取现有文件，合并新发现

必须回写的内容：

| 章节 | 回写内容 | 数据来源 |
|------|---------|---------|
| 典型 BOM 结构 | 子零件树状图（含材料） | README §3 |
| 核心参数与典型范围 | 全部可量化指标及其数值范围 | README §3.2 + 特殊特性 |
| 测试矩阵概要 | 测试类别、样件数、验收标准 | README §5 |
| 常见缺口 | 本次缺失的高优先级文件/信息 | 缺失项跟踪表 |
| 处理注意事项 | 多条件分行场景、易遗漏项 | 特性提取经验 |
| OEM 特有要求 | 按 OEM 分节记录该 OEM 对此零部件的特殊要求 | 全程积累 |

## 步骤 3：组合经验回写

检查 `knowledge/combo/{oem}--{零部件类型}.md` 是否存在：
- **不存在** → 读取 `src/templates/_combo-template.md`，按模板结构创建新文件
- **已存在** → 读取现有文件，合并新发现

必须回写的内容：

| 章节 | 回写内容 | 数据来源 |
|------|---------|---------|
| 项目历史摘要 | 累计处理次数 +1，更新最近处理日期 | 系统日期 |
| 文档组合模式 | 该 OEM 发这类零件时文件包的典型组成 | 文件清单 |
| 典型例外项 | 反复出现的例外事项 | 例外事项清单 |
| 成本参考范围 | 脱敏后的量级参考 | CBS（仅比例，不含绝对值） |
| 历史踩坑记录 | 本组合特有的问题 | 全程积累 |

## 步骤 4：验证

🛑 **HALT — 以下条件全部满足后，项目才可标记为完成：**

- [ ] OEM 知识文件已创建或更新（`knowledge/oem/{oem}.md` 有 diff）
- [ ] 零部件知识文件已创建或更新（`knowledge/component/{type}.md` 有 diff）
- [ ] combo 知识文件已创建或更新（`knowledge/combo/{oem}--{type}.md` 有 diff）
- [ ] 三个文件均不含项目特有数据（零件号、具体日期、具体人名）
- [ ] 三个文件的内容可直接用于下次同类项目

**知识回写未完成 → 禁止关闭项目。必须完成回写。**
